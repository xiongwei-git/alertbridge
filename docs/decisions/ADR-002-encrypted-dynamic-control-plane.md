# ADR-002: 使用服务端渲染后台与 SQLite 加密动态配置

## Status

Accepted — 2026-07-15

## Context

第二阶段需要让非开发者查看投递状态、管理客户端与渠道、轮换密钥、配置路由和静默窗口。目标部署环境是宝塔面板上的单台小型 VPS，仍要求镜像小、内存低、离线可运行，同时不能把敏感凭证明文存入 SQLite 或浏览器。

引入独立 SPA、Node.js 构建链、Redis 或第二个管理服务会扩大镜像、依赖面和运维故障域。只使用静态 JSON 又无法安全地完成在线密钥轮换和即时配置。

## Decision

- 后台由现有 Go 进程使用 `html/template` 服务端渲染，不加载第三方脚本、字体或 CDN 资源。
- 客户端凭证和渠道完整配置使用独立主密钥进行 AES-256-GCM 认证加密后写入 SQLite。
- 管理会话使用随机高熵 Cookie；数据库只保存 SHA-256 令牌哈希。所有已登录写操作使用独立 CSRF Token。
- 动态修改在单个数据库事务完成后构建新的不可变运行快照，并通过原子指针一次切换。
- 新部署不接受 JSON 业务配置种子；客户端、渠道和路由从空状态开始，只通过管理后台写入 SQLite。
- 继续维持单进程、单 SQLite 连接和一个投递 Worker；死信人工重试也复用原队列。

## Consequences

- 生产镜像仍是一个 `scratch` 单二进制，后台不增加运行时服务。
- 主密钥必须与数据库配套备份；主密钥丢失意味着动态凭证不可恢复。
- 单实例配置切换具有一致性，但该设计不支持多实例同时写入；多实例仍是未来独立决策。
- 后台没有客户端 JavaScript，交互以表单和页面跳转为主，但安全策略更简单，弱网环境也更稳定。
