# ADR-009: 轻量通知接口使用 HTTP 200 兼容宝塔

## Status

Accepted — 2026-07-17

## Context

ADR-008 定义的 `POST /api/v1/notifications` 在通知持久化入队后最初返回 `202 Accepted`，用于表达投递由后台 Worker 异步完成。

宝塔自定义消息通道的当前实现只把 HTTP 状态码严格等于 200 视为成功。收到其他 2xx 状态时，它会把正常响应正文显示为“测试信息发送失败”，即使 AlertBridge 已经入队并最终完成投递。这会给管理员造成错误反馈。

## Decision

- `POST /api/v1/notifications` 在认证、校验和持久化入队成功后返回 `200 OK`。
- 成功响应 JSON 保持不变；`outcome=queued` 仍明确表示等待异步投递，而不是外部渠道已经收到消息。
- `POST /api/v1/events` 继续返回 `202 Accepted`，保留完整事件接口原有契约。
- 所有认证、固定路由和级别、请求大小、限流、持久队列、重试与死信边界保持不变。

## Alternatives Considered

- **修改宝塔面板代码接受所有 2xx 状态**：语义更通用，但本地补丁会在面板升级时丢失，且不应要求每个部署者修改第三方系统。
- **通过 Nginx 把 202 改写为 200**：会把 API 兼容逻辑散落到部署配置，增加排障和文档成本。
- **新增宝塔专用路径**：会扩大产品命名的入站接口面，与统一轻量通知契约的边界冲突。
- **保持 202 并接受宝塔误报**：不会破坏 HTTP 语义，但会让主要目标接入场景持续显示错误结果。

## Consequences

- 宝塔保存和测试自定义消息通道时可以正确显示发送成功。
- 只检查状态码的轻量调用方会把“已持久化入队”视为成功；需要确认最终渠道结果时仍必须检查投递记录。
- 已接受任意 2xx 的调用方无需修改；严格要求 202 的轻量调用方需要改为接受 200。
- 完整 HMAC 事件客户端不受影响。
