# ADR-003: 只保留统一事件入站 API

## Status

Accepted — 2026-07-15

## Context

第二阶段曾增加以监控产品命名的 `/hooks/{kind}/{client}` 路径，并用一组常见字段名猜测状态、级别、标题和正文。这些入口没有依据各产品的正式版本契约逐一实现和验证，却对外形成了“原生兼容”的承诺。

启发式转换会在上游字段变化、同名字段语义不同或正文缺失时产生错误事件。它还扩大了不可信输入的解析面，使默认路由、幂等键和签名边界更难解释。AlertBridge 的目标是轻量、稳定和安全的通知网关，不是第三方告警格式转换平台。

## Decision

- `POST /api/v1/events` 是唯一正式、可版本化的入站事件契约。
- 调用方负责将自身数据映射为 AlertBridge 的统一事件模型，并对实际发送的原始 JSON 正文完成 HMAC 签名。
- 服务继续严格拒绝未知字段，并统一执行客户端认证、时间窗、Nonce 防重放、路由授权、限流、幂等和持久化。
- OpenAPI 和管理后台不再公布任何以第三方产品命名的输入路径。
- 已发布过的 `/hooks/*` 路径不再读取正文或验证签名，迁移期固定返回结构化 `410 Gone`，错误码为 `endpoint_removed`，消息指向 `POST /api/v1/events`。

## Alternatives considered

- **维护逐产品解析器**：需要跟踪每个产品的版本、鉴权和测试样例，会扩大依赖面与长期维护成本，不符合当前服务边界。
- **保留通用启发式转换**：实现看似轻量，但无法保证字段语义正确，失败可能表现为错误路由而非明确拒绝。
- **在网关内增加可配置映射规则**：灵活性更高，但会引入表达式执行、配置校验和调试复杂度，暂时没有实际需求支撑。

## Consequences

- 入站攻击面、代码量和测试矩阵变小，API 文档与实际行为保持一致。
- 调用方需要具备格式映射和签名能力；不能直接完成的系统需要来源侧插件或单独部署的可信适配器。
- 旧调用会收到明确、可机器识别的迁移错误，不会被误解析或静默丢弃。
- 未来如确需某一产品适配器，必须基于其正式版本契约、固定样例和认证模型单独决策，不能恢复模糊的通用转换入口。
